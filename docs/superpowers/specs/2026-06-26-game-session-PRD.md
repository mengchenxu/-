# PRD: 游戏会话系统

**日期**: 2026-06-26

---

## Problem Statement

当前 `/骰子` 命令每次返回随机结果后立即结束。用户说"再来一次"时 bot 走 LLM 回复而不是重跑游戏，导致游戏体验割裂。需要游戏会话机制让 bot 进入游戏模式后持续接受游戏命令。

## Solution

新增游戏会话管理器。用户选择游戏后进入"游戏模式"——该群内非 @ 消息被游戏逻辑拦截，直到退出游戏模式。

## User Stories

1. 作为群成员，@bot 选 `/骰子` 后进入游戏模式，后续说"再来一次"不用 @ 也能重掷
2. 作为群成员，说"不玩了/退出/算了"后 bot 退出游戏模式，并以猫娘风格公告
3. 作为群成员，5 分钟不操作自动退出游戏模式
4. 作为群成员，单人开始游戏后，群里其他人也能参与同一个游戏
5. 作为群主，一个群同时只能有一种活跃游戏，bot 不管理多个并发游戏
6. 作为开发者，游戏会话存在 Pipeline 内存中，不持久化，重启清空

## Implementation Decisions

1. **存储**: Pipeline 内存字典 `_game_sessions: dict[str, GameSession]`，key = room_id
2. **数据模型**: GameSession(game_name, started_at, last_input_at)
3. **进入**: @bot + 游戏命令（/骰子）→ 创建 GameSession
4. **命令路由**: 游戏模式中 → 非 @ 消息被游戏处理器拦截，不调 LLM
5. **退出**:
   - 自然退出: 游戏执行完一轮后退出（简单游戏）
   - 手动退出: 用户说"不玩了/退出/算了/不玩"
   - 超时退出: `last_input_at` > 5 分钟
6. **退出公告**: "喵~ 人家退出 [游戏名] 啦！有什么要问的记得 @人家哟 💕"
7. **无冷却**: 退出后立即可以再选游戏

## Testing Decisions

- Mock 游戏处理器测试进入/退出/超时/命令路由
- 不测试真实时间等待

## Out of Scope

- 复杂游戏（猜谜语、接龙等）
- 多游戏并发
- 持久化游戏状态
